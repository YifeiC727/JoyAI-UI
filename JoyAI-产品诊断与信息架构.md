# JoyAI 产品诊断与信息架构重构

---

## 一、现状描述

### 1.1 当前一级导航

```
侧边栏
├── + 新对话
├── 文字助手
├── 视觉解读
├── 视觉创作
└── 语音交互
```

### 1.2 各功能模块详细描述

#### 文字助手

- **交互形式**：标准聊天界面（Chat UI）
- **核心能力**：内容创作与改写、逻辑分析、编程与技术、学习与辅导、产品与运营、日常事务
- **历史记录**：有，按对话分组
- **附件支持**：无

#### 视觉解读

- **交互形式**：聊天界面（Chat UI），支持上传图片/视频附件
- **核心能力**：图片/视频内容理解与解读（VL 模型）
- **历史记录**：有，按对话分组
- **附件支持**：图片、视频
- **隐藏入口**：聊天框右侧小圆点 → 跳转至「实时识别（Interaction Model）」
  - 独立页面，三栏布局：左侧新建对话 | 中间视频/直播流 | 右侧文字交互区
  - 能力：实时视频/直播画面分析，如人物计数、场景描述等
  - 交互模式：Streaming（持续输入 → 实时输出）

#### 视觉创作

- **交互形式**：聊天界面（Chat UI），通过文字描述生成图片/视频
- **核心能力**：文生图、文生视频
- **历史记录**：无独立对话概念，所有生成内容（图片+视频）混在同一记录中，无法新建对话
- **附件支持**：无
- **隐藏入口**：聊天框右侧小圆点 → 跳转至「Echo（长视频生成器）」
  - 独立页面，三栏布局：左侧新建对话 | 中间 AI 聊天（含剧本/风格确认） | 右侧创作面板
  - 工作流：策划剧本 → 分镜脚本 → 生成分镜 → 合成成片
  - 交互模式：Workflow（多步骤线性流程）

#### 语音交互

- **交互形式**：非聊天界面，顶部横排三个子功能 Tab
- **核心能力**：
  - **Talker**：语音聊天（实时语音对话）
  - **Voice**：音色创作（自定义语音音色）
  - **多角色配音**：多角色语音合成
- **历史记录**：不明确
- **附件支持**：不明确

### 1.3 实际产品能力全景

当前 JoyAI 实际包含 **8 个独立产品能力**，但 UI 上仅呈现为 4 个一级入口：

```
JoyAI 实际产品树
├── 文字助手 .............. 一级入口可见
├── 视觉解读 .............. 一级入口可见
│   └── 实时识别 .......... 隐藏在小圆点中
├── 视觉创作 .............. 一级入口可见
│   └── Echo 长视频 ....... 隐藏在小圆点中
└── 语音交互 .............. 一级入口可见
    ├── Talker ............ 二级 Tab
    ├── Voice ............. 二级 Tab
    └── 多角色配音 ........ 二级 Tab
```

## 二、核心问题诊断

### 问题 1：导航层级混乱（严重度：P0）

**现象**：「动作」「功能」「产品」三种不同维度被放在同一层导航。

```
当前导航            实际属性
─────────────────────────────
+ 新对话            → 动作（Action）
文字助手            → Agent
视觉解读            → Agent
视觉创作            → Agent
语音交互            → 功能集合（Feature Group）
```

类比：相当于把「新建文件 / Word / Excel / PowerPoint」放在同一层 —— 「新建」不是软件。

**影响**：
- 用户无法建立一致的心智模型
- 「新对话」脱离具体 Agent 存在，用户不知道新建的对话属于哪个功能
- 语音交互本身不是单一功能，而是三个独立工具的容器

---

### 问题 2：产品能力被隐藏，可发现性差（严重度：P0）

**现象**：8 个产品能力中，有 2 个（实时识别、Echo）藏在聊天框旁的小圆点里。

```
视觉解读页面
┌──────────────────────────────────┐
│                                  │
│         聊天区域                  │
│                                  │
│  ┌────────────────────────┐  ●  │ ← 这个圆点通往「实时识别」
│  │  输入框               │      │
│  └────────────────────────┘      │
└──────────────────────────────────┘
```

**影响**：
- 用户完全不知道圆点的存在和用途
- 没有任何文字提示说明点击后去哪里
- Hidden Entry 违反 UI 设计基本原则：用户不应该需要「发现」核心功能
- 两个完整的独立产品（实时识别、Echo）的入口被降级为一个无标签的小圆点

---

### 问题 3：四种交互模式被强塞进同一种 UI（严重度：P1）

**现象**：产品实际存在四种完全不同的交互模式，但几乎全部使用 Chat UI 承载。

| 交互模式 | 特征 | 对应功能 | 当前 UI |
|---------|------|---------|--------|
| **Chat** | 一问一答，异步 | 文字助手、视觉解读、视觉创作 | 聊天界面 |
| **Workflow** | 多步骤线性流程，有阶段 | Echo 长视频 | 聊天界面 + 右侧面板 |
| **Streaming** | 持续输入、实时输出 | 实时识别 | 视频 + 文字区 |
| **Tool** | 工具型，操作导向 | Talker、Voice、多角色配音 | Tab + 工具面板 |

**影响**：
- Chat UI 无法很好地承载 Workflow（Echo 的剧本→分镜→生成流程在聊天框里非常拧巴）
- Chat UI 无法承载 Streaming（实时识别需要视频流 + 持续输出，不是一问一答）
- 语音工具硬要做成对话形态反而增加认知负担
- 不同模式的 UI 混用，用户需要反复适应

---

### 问题 4：历史记录逻辑不一致（严重度：P1）

**现象**：各功能的历史记录/对话管理方式完全不统一。

| 功能 | 有历史记录 | 可新建对话 | 对话模型 |
|------|-----------|-----------|---------|
| 文字助手 | 有 | 有 | 标准 |
| 视觉解读 | 有 | 有 | 标准 |
| 视觉创作 | 无 | **不可** | 所有内容混在一起 |
| 实时识别 | 有 | 有 | 标准 |
| Echo | 有 | 有 | 项目制 |
| Talker | 不明确 | 不明确 | 不明确 |
| Voice | 不明确 | 不明确 | 不明确 |
| 多角色配音 | 不明确 | 不明确 | 不明确 |

**影响**：
- 视觉创作没有对话概念，所有图片/视频堆在一起，无法组织和回溯
- 用户在不同功能间切换时，对「历史在哪里」的预期完全不一致
- 没有统一的 Conversation 数据模型，导致无法做跨功能的统一历史管理

---

### 问题 5：「新对话」入口设计反直觉（严重度：P1）

**现象**：「新对话」与四个功能并列在一级导航，但实际创建的对话归属于当前所在的功能。

**影响**：
- 用户在不进入具体功能时，不知道「新对话」会创建哪种类型的对话
- 「新对话」本质是一个 Action，不应与 Agent/Feature 并列
- 正确做法：每个 Agent/Feature 内部提供 `+ 新建对话` 按钮（ChatGPT / Claude / Gemini 的通用做法）

---

## 三、核心结论

### 根本原因

> 不是某一个页面不好，而是**整个信息架构（Information Architecture）混乱**。

JoyAI 当前的问题本质上是：

1. **8 个产品被压缩成 4 个入口**，导致功能被隐藏
2. **4 种交互模式被统一成 Chat UI**，导致体验割裂
3. **没有统一的 Conversation 模型**，导致数据管理混乱
4. **动作（新对话）和功能（Agent）混在同一层级**，导致导航反直觉

---

## 四、产品重新定位

### 4.1 核心认知转变

> JoyAI 不是一个聊天机器人，而是一个 **AI 能力平台（AI Capability Platform）**。

这个定位转变决定了所有后续设计决策：

| 维度 | 聊天机器人思路 | AI 能力平台思路 |
|------|--------------|---------------|
| **首要目标** | 让用户快速开始对话 | 让用户一眼看清所有能力 |
| **导航逻辑** | 隐藏复杂度，只露一个入口 | 展示能力版图，结构化呈现 |
| **核心受众** | 终端用户（追求简单） | 决策者 + 终端用户（需要看到全貌） |
| **成功标准** | 用户能完成任务 | 用户理解能力范围 + 能完成任务 |

### 4.2 目标用户的核心问题

老板/决策者进入 JoyAI，脑子里的问题是：

> 1. JoyAI **到底有什么能力**？
> 2. 每个能力之间是**什么关系**？

而**不是**：

> 我今天该点哪个聊天框？

所以 IA 首先要回答的是 **能力地图（Capability Map）**，而不是对话入口。

### 4.3 为什么保留四大能力域

不同于 To C 产品（如 ChatGPT）把一切塞进一个聊天框来降低复杂度，JoyAI 作为能力展示平台，反而需要让能力**可见、可枚举、有结构**。

保留四大一级分类：**文字助手 / 视觉理解 / 视觉创作 / 语音能力**

原因：
- 符合老板对 AI 能力版图的认知（一眼知道：文字、视觉、创作、语音都覆盖了）
- 四个领域相互独立，边界清晰
- 比三级分类（助手 / 创作 / 语音）更能体现能力的广度和深度
- 比 ChatGPT 式「一个框」更有展示价值

## 五、信息架构重设计

### 5.1 三层架构原则

整个 JoyAI 的 IA 遵循三层结构，每层职责明确：

```
┌─────────────────────────────────────────────────────────────────┐
│  第一层：能力域（Capability Domain）                              │
│  回答：JoyAI 有哪几大类能力？                                     │
│  例如：文字助手 │ 视觉理解 │ 视觉创作 │ 语音能力                    │
├─────────────────────────────────────────────────────────────────┤
│  第二层：具体能力（Capability）                                    │
│  回答：这个领域下有哪些具体能力？                                   │
│  例如：视觉理解 → 图片理解 / 视频理解 / 实时识别                    │
├─────────────────────────────────────────────────────────────────┤
│  第三层：交互界面（Interaction）                                   │
│  回答：这个能力用什么方式操作？                                     │
│  例如：实时识别 → Streaming UI（视频流 + 实时文字）                  │
└─────────────────────────────────────────────────────────────────┘
```

**核心原则：能力与交互解耦**
- 老板看到的是清晰的 AI 能力地图（第一层 + 第二层）
- 用户进入后看到的是最适合该能力的交互方式（第三层）
- 不同的交互模式（Chat / Studio / Live / Tool）不会让导航变混乱，因为它们只在第三层出现

### 5.2 完整信息架构树

```

│   │
│   ├── 视频理解
│   │   ├── [交互方式] Chat（支持视频上传）
│   │   └── [历史管理] 对话列表 + 新建对话
│   │
│   └── 实时识别（Live Vision）
│       ├── [交互方式] Streaming（视频流 + 实时输出）
│       └── [历史管理] 会话列表 + 新建会话
│
├── 视觉创作
│   │
│   ├── 快速创作
│   │   ├── AI 图片
│   │   │   ├── [交互方式] Chat / Gallery

│   │   └── AI 视频

│   │       └── [历史管理] 对话列表 + 新建对话
│   │
│   └── 专业创作
│       └── Echo Studio
│           ├── [交互方式] Workflow（剧本 → 分镜 → 生成 → 成片）

    ├── Talker 语音聊天

    │
    ├── Voice 音色创作
    │   ├── [交互方式] Tool（上传样本 → 生成音色）
```

### 5.3 关键变化：隐藏能力全部提升为正式入口

#### 变化一：实时识别从隐藏圆点变为一级可见能力

```
改版前                              改版后
─────────────────────               ─────────────────────
视觉解读                            视觉理解
┌────────────────┐                  ├── 图片理解
│   聊天区域      │                  ├── 视频理解
│           ● ←─┼── 藏在这里        └── 实时识别（Live Vision）
│   输入框       │                        ↑
└────────────────┘                        正式入口，明确可见
```

#### 变化二：Echo 从隐藏圆点变为品牌级入口

```
改版前                              改版后
─────────────────────               ─────────────────────
视觉创作                            视觉创作
┌────────────────┐                  ├── 快速创作
│   聊天区域      │                  │   ├── AI 图片
│           ● ←─┼── 藏在这里        │   └── AI 视频
│   输入框       │                  └── 专业创作
└────────────────┘                      └── Echo Studio ⭐
                                              ↑
                                         品牌级入口
                                         JoyAI 最强能力，不应隐藏
```

**Echo 应该成为一个品牌**，而不是一个彩蛋。它是整个产品最具技术壁垒的能力，老板看到后会认为这是核心竞争力，而目前的设计让它完全不可见。

#### 变化三：语音三个子功能获得平等入口

```
改版前                              改版后
─────────────────────               ─────────────────────
语音交互                            语音能力
[Talker] [Voice] [配音]             ├── Talker 语音聊天
    ↑                               ├── Voice 音色创作
    挤在顶部 Tab 里                   └── 多角色配音
    没有结构感                              ↑
                                         清晰并列，各有独立入口
```

### 5.4 左侧导航设计：AI 能力树

左侧导航不再是普通菜单，而是一棵 **AI 能力树（Capability Tree）**。

```
┌──────────────────────────────┐
│  🤖 JoyAI                    │
│                              │
│  ▼ 文字助手                   │
│      + 新建对话               │
│      Chat 1                  │
│      Chat 2                  │
│                              │
│  ▼ 视觉理解                   │
│      ▸ 图片理解               │
│      ▸ 视频理解               │
│      ▸ 实时识别               │
│                              │
│  ▼ 视觉创作                   │
│    ▸ 快速创作                 │
│      ├ AI 图片                │
│      └ AI 视频                │
│    ▸ Echo Studio             │
│                              │
│  ▼ 语音能力                   │
│      ▸ Talker                │
│      ▸ Voice                 │
│      ▸ 多角色配音             │
│                              │
└──────────────────────────────┘
```

**能力树的优势**：
- **几秒看懂全貌**：老板不需要逐个点击，展开树即可看到 JoyAI 的完整能力版图
- **层级关系清晰**：一级是能力域，二级是具体能力，不再混乱
- **专业感**：树状结构天然带有技术平台的质感，比扁平菜单更有架构感
- **可扩展**：未来新增能力只需在对应节点下加叶子，不改变整体结构
- **「新对话」回归正确位置**：每个能力节点内部的 `+` 按钮，而非全局一级入口

- 整体传达的信息是：这是一个完整的 AI 能力平台，不是一个简单的聊天工具

## 六、设计原则总结

### 原则 1：能力可见（Capability Visibility）

> 所有能力都有明确入口，不允许隐藏入口（Hidden Entry）。

Echo 是核心竞争力，不是彩蛋。实时识别是高级能力，不是附属功能。

### 原则 2：能力与交互解耦（Capability-Interaction Decoupling）

> 导航展示的是「能力」，进入后呈现的是「交互」。

老板看到的是能力地图：文字 / 视觉理解 / 视觉创作 / 语音。
用户操作的是最优交互：Chat / Streaming / Workflow / Tool。
两者不应混在一起。

### 原则 3：三层清晰（Three-Layer Clarity）

> 一级 = 能力域，二级 = 具体能力，三级 = 交互界面。

任何新增功能只需回答三个问题：
1. 它属于哪个能力域？（决定放在哪棵子树下）
2. 它是什么能力？（决定导航中的名称和位置）
3. 它用什么交互？（决定进入后的界面形态）

### 原则 4：历史统一（Unified History）

> 所有能力共享统一的 Session 模型，只是 Session 类型不同。

不再出现「有的功能有历史，有的没有」的割裂体验。
